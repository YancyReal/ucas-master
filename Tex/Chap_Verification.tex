\chapter{系统验证方法与实验平台}\label{chap:verification}

\section{实验环境与软硬件平台}

为验证 LoongArch 平台 ART 执行引擎适配结果的正确性与可用性，本文在真实目标设备上建立了
完整的编译、部署、回归与 benchmark 实验链路。整体实验平台基于龙芯 3A6000 处理器，采用
四核、主频 2.5GHz、16GB 内存的硬件配置，目标系统为 Android 15，内核版本为 Linux 6.6.46。
软件侧以 AOSP 15 源码中的 \texttt{art/} 子项目为研究对象，围绕 LoongArch64 平台补齐执行引擎
相关模块，并在此基础上开展后续的正确性验证与性能分析。

本文在验证阶段统一采用 LoongArch64 的 64 位调试构建口径。一方面，调试构建更便于暴露执行引擎
适配过程中潜在的调用约定错误、慢路径跳转错误和异常恢复问题；另一方面，本文后续三项正式性能方案
也建立在相同的调试构建环境之上，因此在验证章节保持同一执行口径，有助于保证论文前后实验环境的一致性。

需要指出的是，本文关注的是 LoongArch 平台 ART 执行引擎的系统适配与执行行为，而非单个组件的
孤立验证。因此实验平台的组织原则是：所有验证步骤均围绕“同一套编译产物、同一台目标设备、同一条
执行链路”展开，避免由于版本漂移或部署对象不一致而导致结果失真。

\section{编译、部署与测试流程}

\subsection{编译流程}

ART 的平台适配工作涉及解释器、JIT 编译器、JNI 编译器与运行时入口点等多个子模块，因此本文在
功能验证和性能实验前均优先执行完整 ART 编译，以确保所有相关动态库、桩代码和目标机测试产物处于
同一源码版本下。为避免将作者个人工作站路径误写为统一实验条件，本文用 \texttt{AOSP\_ROOT}
表示实验所使用的 AOSP 15 源码树根目录；在任意复核环境中，只需将其替换为本地源码路径即可沿用
同一编译流程。本文采用的统一编译入口可写为：

\begin{lstlisting}[language=bash,basicstyle=\ttfamily\small]
cd ${AOSP_ROOT} && art/tools/buildbot-build.sh --target
\end{lstlisting}

采用统一编译入口的原因有两点。首先，ART 的许多平台相关问题不会只出现在单个动态库中，而是会同时
影响 \texttt{libart.so}、\texttt{libartd-compiler.so}、测试用运行时环境以及若干目标机辅助组件；
若仅进行局部增量编译，容易出现宿主侧与目标机侧产物不一致的问题。其次，完整编译能够在较早阶段暴露
链接错误、未实现符号和部分配置不匹配问题，从而降低后续部署与测试阶段的干扰项。

\subsection{部署流程}

完成编译后，本文默认采用轻量部署方式将新的 ART 产物推送到目标设备。需要说明的是，部署步骤与
设备分区挂载方式、构建输出目录以及调试权限配置紧密相关，因此正文不再把作者本地维护的 host 侧
同步脚本写成“统一命令”，而只保留其对实验结论真正重要的流程约束：从本次构建输出中收集 ART
相关库、可执行文件和测试产物，再通过 \texttt{adb push}、\texttt{adb sync} 或等价同步机制
覆盖到目标设备对应路径，最后在相同设备状态下执行回归与 benchmark。

轻量部署的核心目标是只同步与 ART 执行链路直接相关的关键产物，减少完整系统镜像推送带来的额外时间
成本与环境扰动。只有在明确涉及系统镜像、框架依赖或其他重量级组件同步时，才使用全量推送方式。对本文
而言，大多数执行引擎适配和优化实验均可通过轻量部署完成，这也使得“修改、编译、推送、验证”的迭代
周期保持在可接受范围内。

需要强调的是，目标机回归和 benchmark 均建立在部署后的真实设备环境之上。若未先推送新产物，随后执行
\texttt{art/test.py} 或 benchmark 脚本得到的结果将无法反映当前源码状态。因此，本文在实验链路上
始终遵循“编译、推送、目标机回归、benchmark”这一固定顺序。

\subsection{验证链路组织方式}

本文将所有验证过程组织为四级链路：

\begin{enumerate}
    \item \textbf{编译校验}：确认平台相关代码在当前源码状态下能够完整构建。
    \item \textbf{部署同步}：将新编译产物推送至目标设备，保证待测环境与源码版本一致。
    \item \textbf{目标机回归}：执行最小相关 run-test 或功能性测试，验证语义正确性与稳定性。
    \item \textbf{benchmark 评测}：在已确认正确的版本上采集性能数据、日志和火焰图。
\end{enumerate}

这种组织方式的价值在于将“功能正确”和“性能可比”严格分离。只有在前一层验证通过后，下一层实验结论
才具有解释价值。特别是在执行引擎适配与性能优化并行推进的过程中，这一链路可以避免把由语义错误引发的
崩溃、回退或异常抖动误判为性能问题。

\section{正确性验证方法}

\subsection{Host侧测试}

Host 侧测试的作用，是在不依赖目标设备的前提下尽早验证 ART 基础功能是否受到平台化改动影响。本文主要
使用 ART 自带的 Host GTest 与相关单元测试集，覆盖 DEX 文件处理、命令行解析、编译器基础设施、
运行时公共库等子系统。此类测试虽然不能直接证明 LoongArch 目标机上所有执行路径都正确，但能够有效筛查
平台适配过程中引入的通用性回归，例如 DEX 验证器行为变化、编译器公共组件失效或命令行参数处理异常。

从作用定位上看，Host 侧测试更接近“第一道过滤器”。它不能替代目标机验证，但可以在进入设备侧实验前
提前排除一批明显错误，从而减少目标机调试的复杂度。

\subsection{Target侧run-test验证}

Target 侧 run-test 是本文验证执行引擎语义正确性的核心手段。ART 测试框架会将测试用例编译为 DEX，
并把动态库、测试程序与运行脚本组织到设备侧测试沙箱中执行，从而覆盖解释执行、编译执行、异常处理、
对象访问和调用约定等关键行为。

在日常定位中，本文优先采用最小相关测试集合进行回归。例如，对执行引擎和 JIT 相关改动，常用命令为：

\begin{lstlisting}[language=bash,basicstyle=\ttfamily\small]
./art/test.py -r --target --no-prebuild --debug --no-image --64 --jit -j 1 -t <case>
\end{lstlisting}

当需要验证平台适配阶段的整体正确性时，则使用更大范围的目标机测试集合，分别在 C++ 解释器、Nterp
解释器和 JIT 编译器三种模式下执行。根据当前已整理材料，三种执行模式下均完成了 1033 个测试用例的验证，
通过情况如表~\ref{tab:target-run-test-result} 所示。

这一结果说明，LoongArch 平台 ART 执行引擎已经不仅能够运行简单样例，而且在三条主要执行路径上均通过了
系统性的目标机回归，具备进入 benchmark 评测阶段的语义基础。

\subsection{libcore测试}

仅依赖 run-test 仍不足以证明 Java 核心类库层面的行为完全正确，因此本文进一步引入 libcore 测试集，
用于覆盖集合框架、输入输出、并发、数学运算、加解密与网络等 Java API 语义。本文采用设备侧运行方式执行
libcore 测试，并验证全部 146 个测试包。结果表明，相关测试包均通过，部分因架构差异自动跳过的用例不影响
主要语义结论。

与 run-test 相比，libcore 测试的意义在于：前者更贴近运行时和执行引擎行为，后者则更贴近 Java 应用
层面对 API 语义的观察结果。两者结合后，才能较完整地证明 LoongArch 平台 ART 已具备作为 Android
应用运行时的基本正确性。

\subsection{差分测试与辅助调试方法}

在执行引擎适配过程中，单纯依赖“通过或失败”的测试结果往往不足以快速定位错误来源，因此本文还建立了
若干辅助调试方法。

首先，针对解释器路径，本文采用差分测试方法，将同一段 Java 程序分别在参考平台和 LoongArch 平台的
解释执行环境中运行，通过追踪每条字节码执行后的寄存器与对象状态，定位首次出现偏差的 DEX PC。该方法
尤其适用于对象访问、字段读写和类型检查等语义细节问题的定位。

其次，针对 Nterp，本文采用逐类指令递增验证策略。即在尚未补齐全部 opcode 前，先在运行时中过滤未实现
指令，仅对已实现指令类别开放 Nterp 路径；每新增一组指令，即与 C++ 解释器结果进行对照，缩小错误定位
空间。这种方法有效降低了汇编解释器一次性大规模实现带来的调试难度。

再次，针对 JIT 后端，本文结合 CFG 导出、反汇编输出和运行时日志进行定位。通过查看控制流图中 HIR 节点、
块间跳转和最终生成的 LoongArch64 汇编，可以较快判断问题是发生在前端图构建、后端指令选择，还是运行时
入口点调用阶段。

\section{benchmark设计与正式baseline建立}

\subsection{benchmark选择原则}

性能实验部分的目标，不是给出“泛化到所有 Android 应用”的绝对结论，而是在统一实验口径下观察执行引擎
相关优化对典型工作负载的影响。基于这一目标，本文选择 DaCapo 与 SPECjvm2008 作为主要 benchmark
集合\citep{Blackburn2006,SPECjvm2008}。前者更偏向现代 Java 应用中的对象分配、字符串处理与复杂
控制流行为；后者则包含更靠近经典 JVM 计算与数值处理路径的负载。

与此同时，本文在 benchmark 结果解释上明确参考统计严谨的 Java 性能评测方法与 warmup 行为研究，
避免将单轮波动或暂时性升降误判为稳定收益\citep{Georges2007,Barrett2017Warmup}。对于基准覆盖范围
本身，本文也有意识地将结论约束在当前 workload 边界内，而不将其直接外推为所有现代 JVM 应用的普遍
规律；这与 Renaissance 等后续 benchmark 工作对“传统套件覆盖有限”的提醒是一致
的\citep{Prokopec2019Renaissance}。

在具体子项上，本文在正式图表中保留五个 workload，分别为：
\begin{itemize}
    \item \texttt{DaCapo/pmd}
    \item \texttt{SPECjvm2008/mpegaudio}
    \item Scimark 子项：\texttt{lu.small}、\texttt{sor.small} 和 \texttt{fft.small}
\end{itemize}
其中，
\texttt{pmd} 用于观察解释器驻留、JIT 触发时机与字符串运行时热点；\texttt{mpegaudio}
用于提供与 \texttt{pmd} 明显不同的短生命周期吞吐 workload；\texttt{lu.small} 是本文用于验证
LoongArch64 LSX 浮点 SIMD 方案的主要受益对象；\texttt{sor.small} 与 \texttt{fft.small}
则用于观察该类数值优化在相邻 workload 上的可迁移性与副作用边界。通过这样的五 workload 组合，
本文能够避免把某一类局部收益误写为“适用于所有 benchmark”的普遍结论。

\subsection{正式baseline口径}

为避免性能结论受到设备状态、命令差异或临时试跑数据的干扰，本文对 benchmark 实验采用固定口径，
并明确区分正式单次比较数据与重复测量验证数据。

\begin{itemize}
    \item \texttt{2026-03-13 final-comparison}：用于正文主表和主图的正式单次对比口径。
    同日 baseline 为：\texttt{pmd=3272\,msec}、\texttt{mpegaudio=25.38\,ops/m}、
    \texttt{lu.small=46.22\,ops/m}、\texttt{sor.small=24.76\,ops/m} 和
    \texttt{fft.small=76.43\,ops/m}。三项单方案结果均相对这组 baseline 计算。
    \item \texttt{2026-03-14 statistical-validation}：用于重复测量验证的独立数据集，
    仅检验方案一在 \texttt{pmd} 上的收益方向和方案三在 \texttt{lu.small} 上的稳定收益，
    其结果单独展示，不与前一日的正式单次结果混算。
\end{itemize}

其中，\texttt{DaCapo} 的评价指标为执行耗时，单位为 \texttt{msec}，数值越小表示性能越好；
\texttt{SPECjvm2008} 的评价指标为 \texttt{ops/m}，数值越大表示吞吐越高。本文在后续分析中
始终遵循这一指标方向，不将不同 benchmark 的分值含义混为一谈。

需要说明的是，上述 \texttt{final-comparison} 与 \texttt{statistical-validation}
目录中的 \texttt{run.log}、\texttt{perf.data}、\texttt{report\_symbol.txt}、
\texttt{report\_callgraph.txt} 和 \texttt{flamegraph.svg} 等原始产物属于作者实验归档，
用于论文写作、交叉核验与历史记录；考虑到仓库提交范围与数据体积限制，当前论文仓库未附带完整
\texttt{test-data/} 目录。因而，论文正文中出现的正式性能结果若无特殊说明，均指上述两类固定实验
口径及其导出的分数与热点统计，而不是某个随仓库公开发布的目录快照。

\subsection{性能数据采集方式}

为保证性能分析不仅停留在单一分数层面，本文在 benchmark 运行时同步采集原始日志与热点数据。每次正式实验
目录中通常保留如下文件：

\begin{itemize}
    \item \texttt{run.log}：记录 Host 调用命令、解析后的 \texttt{dalvikvm64} 命令与远端 shell payload；
    \item \texttt{perf.data}：采集得到的原始采样数据；
    \item \texttt{report\_symbol.txt}：符号级热点汇总；
    \item \texttt{report\_callgraph.txt}：调用图视角的热点信息；
    \item \texttt{flamegraph.svg}：火焰图，可直观反映热点路径分布。
\end{itemize}

通过同时保留分数、日志与 profile 数据，本文在后续优化分析中可以建立更完整的因果链，即从“分数变化”
进一步追溯到“热点分布变化”和“固定成本变化”，从而避免仅凭单次跑分对方案优劣做出过度推断。
这种“尽量降低观测扰动、同时保留足够因果证据”的思路，也与 Android 场景下低扰动性能调试研究的
方法论相吻合\citep{Luo2022Hubble}。

\section{有效性威胁与实验控制}

尽管本文已经通过固定编译、固定部署顺序、目标机最小相关回归以及统一 baseline 归档来控制实验变量，
但面向 JIT 运行时的性能研究仍然不可避免地受到若干有效性威胁影响。为避免在第 5 章中对实验结论做出
超出证据边界的推断，本文在写作中对以下因素予以明确约束。

\begin{enumerate}
    \item \textbf{设备状态与后台负载扰动。} benchmark 结果会受到目标机温度、后台服务活动、
    CPU 频率波动和设备前序运行历史的影响。因此，本文不混用跨日期、跨设备状态的 baseline，而是将
    `2026-03-13` 的正式单次对比数据与 `2026-03-14` 的重复测量数据严格分离，避免把环境漂移误写成
    方案收益。
    \item \textbf{\texttt{debug} 构建口径限制。} 本文的正式实验统一建立在 LoongArch64
    \texttt{debug} ART 环境下。这一口径有利于在移植收敛阶段保留更多调试信息与验证能力，但也会放大
    JIT 调试信息、符号处理和部分运行时固定成本。因此，正文中所有性能结论均被限制为“固定
    \texttt{debug} baseline 下的阶段性结果”，不直接外推到 release 构建或其它系统配置。
    \item \textbf{重复测量样本数有限。} 本文在第 5 章仅对最关键的两条性能主张进行了有限轮次的同日
    重复测量，其目的在于区分“可稳定复现的收益”与“仍需继续验证的收益方向”，而不是进行严格统计显著性
    检验。因此，论文中采用原始散点、均值和标准差来表达结论强度，而不滥用显著性术语。
    \item \textbf{JIT 符号可观测性有限。} 部分实验归档中存在 \texttt{perf-*.map} 缺失或 JIT Java
    符号不完整的情况，这意味着火焰图和 \texttt{report\_symbol.txt} 更适合用于判断“热点路径是否被
    命中”和“固定成本是否相对变化”，而不适合支撑过于细粒度的 Java 方法级归因。因而，本文在第 5 章中
    始终将分数变化、回归结果和 profile 证据结合解释，而不单独依赖某一类可观测结果下结论。
\end{enumerate}

\section{验证结果汇总}

\subsection{移植阶段验证结果}

结合第 3 章的实现工作和本章给出的验证链路，可以将 LoongArch 平台 ART 执行引擎适配阶段的结果概括为
三点。

第一，Host 侧测试验证了平台适配未破坏 DEX 处理、命令行解析和运行时公共组件等基础功能，为设备侧回归
提供了前置保证。

第二，Target 侧 run-test 在 C++ 解释器、Nterp 和 JIT 三种执行模式下均通过了 1033/1033 的测试集，
说明 LoongArch 平台执行引擎的三条主要执行路径在语义层面已经收敛。

第三，libcore 测试集的通过表明，LoongArch 平台 ART 不仅在运行时内部测试上具备正确性，也能够支撑
Java 核心类库层面的实际 API 语义。

这三类结果共同表明：第 3 章所建立的执行引擎能力已经从“可运行”推进到“可系统验证”，为后续性能研究
提供了可靠基础。

\subsection{优化阶段验证要求}

性能优化阶段并不重复执行全量系统验证，而是遵循“最小相关回归优先”的原则。即每个候选优化方案在进入
benchmark 实验前，先根据改动所在模块选择直接受影响的目标机测试集合。例如，字符串热路径优化会优先验证
字符串相关 checker 测试和 JIT 回归；JIT 参数调整方案则优先验证 JIT 缓存、JNI stub 和编译相关用例。

这一做法的目的，是在维持验证效率的同时保证结论可信度。若最小相关测试未通过，则 benchmark 结果不具有
解释价值；只有在相关回归通过后，后续性能差异才可以归因于方案本身，而不是功能错误或未定义行为。

\section{本章小结}

本章从实验环境、编译部署链路、正确性验证方法、benchmark 组织方式以及正式 baseline 口径等方面，
建立了本文后续性能研究所依赖的统一实验框架。与第 3 章偏重模块实现不同，本章强调的是“如何证明这些
实现是可信的”，并为下一章的优化实验提供统一的数据来源、评价标准和解释边界。
