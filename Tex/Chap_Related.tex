\chapter{相关技术与问题分析}\label{chap:related}

\section{ART运行时体系结构}

Android Runtime（ART）是 Android 平台上负责 Java/Kotlin 应用执行的核心运行时，其职责并不
局限于“解释执行字节码”，而是贯穿应用装载、字节码校验、解释执行、即时编译、安装期编译、垃圾回收、
JNI 桥接与异常处理等完整链路\citep{AndroidRuntime2024,Ghuloum2014}。从源码组织上看，
\texttt{runtime/} 负责线程、对象模型、类链接、垃圾回收与运行时服务，\texttt{interpreter/}
负责解释执行路径，\texttt{compiler/} 与 \texttt{dex2oat/} 负责编译基础设施，
\texttt{libdexfile/} 负责 DEX 文件解析与校验，多个子模块共同构成一个分层协同的执行系统。

与传统“单一解释器”式虚拟机不同，ART 的一个显著特点是执行模式并存。应用方法在不同阶段可能经过
DEX 校验、解释执行、baseline JIT 编译、优化重编译以及安装期 AOT 编译等多条路径；运行时则负责
在这些路径之间维护统一的对象语义、异常语义和线程状态\citep{ArtJitCompiler2024,Arnold2005}。
因此，对 ART 的平台适配不能简单理解为某个独立模块的移植，而必须把解释器、编译器、运行时入口点、
调用约定和调试支撑作为一个整体来看待。

从 Android 应用的实际执行链路看，ART 既要保证冷启动阶段的可用性，又要保证热点路径上的执行效率。
这种双重目标决定了 ART 的工程组织方式天然具有层次性：以解释器和运行时为最小可运行闭环，以 JIT
和 AOT 作为进一步提升吞吐和启动性能的高性能路径，以 Intrinsic、快速入口点和 JNI 桥接作为连接
Java 语义与机器级实现的关键补充。对本文而言，理解这种层次结构，是分析 LoongArch 平台适配范围与
后续优化边界的前提。

\section{ART的执行与编译机制}

\subsection{DEX字节码与寄存器虚拟机}

ART 的上层执行对象是 DEX 字节码。与传统 JVM 栈式字节码不同，DEX 采用寄存器虚拟机模型，
字节码指令通常显式给出源寄存器与目标寄存器，从而减少中间入栈、出栈操作，提高指令表达密度和移动端
执行效率\citep{Bornstein2008,Shi2008}。这种设计有利于降低解释器中的栈操作频率，也为后续的
SSA 构图、寄存器分配和机器码生成提供了更直接的中间表示基础。

寄存器虚拟机模型并不意味着平台适配工作更简单。相反，它要求解释器、JIT 编译器和运行时在寄存器槽位、
ShadowFrame、方法调用参数区以及返回值位置之间保持一致映射。一旦平台相关调用约定、对象访问语义或
异常恢复路径处理不当，DEX 层面看似简单的字段读写、类型检查和方法调用就会在目标机上表现为寄存器
污染、栈帧破坏或错误异常传播。因此，DEX 的寄存器化设计一方面提高了执行潜力，另一方面也要求平台
适配必须在执行语义和底层调用约定之间建立严格对应关系。

\subsection{C++解释器、Nterp与解释器优化机制}

解释器是 ART 最基础、也是最具可移植性的执行路径。C++ 解释器依赖平台无关的
\texttt{switch-case} 主循环实现字节码分派，便于在新架构上较早建立“可执行、可调试、可验证”的
最小闭环。然而，从解释器性能研究的经典结论可知，解释执行的瓶颈往往不仅来自单条指令实现本身，也来自
分派结构和间接分支预测行为\citep{Ertl2003}。这也是为什么 ART 在 C++ 解释器之外，又引入了
Nterp 这类面向热点解释执行的汇编解释器路径。

Nterp 的核心思想是通过预构建派发表和固定寄存器协议，将 opcode 直接映射到对应的汇编处理块，从而
减少传统 \texttt{switch-case} 分派中的共享分支开销。其设计与高效解释器研究中的 threaded code
思路是一致的：通过让不同指令拥有更明确的控制流上下文，提高处理器对分派路径的可预测性\citep{Ertl2003}。
在此基础上，quickening、inline cache 等解释器优化技术则进一步说明，解释器的优化空间并不只在分派
层面，还可以体现在“把常见慢路径提前特化”为更便宜的执行形态\citep{Brunthaler2010Quickening,
Brunthaler2010InlineCaching}。

对本文的研究对象而言，这些结论具有直接启发意义。LoongArch 平台 ART 的适配不应仅以“能否运行解释器”
为终点，而应把解释器路径视为整个执行引擎的语义基线与性能基线。只有先建立与 C++ 解释器一致的语义，
再逐步补齐 Nterp 的入口、帧布局、寄存器约束和字节码处理逻辑，后续 JIT、OSR 和运行时协同路径才有
稳定的参照系。

\subsection{JIT、AOT与分层编译机制}

在解释执行之上，ART 通过 JIT 与 AOT 形成混合执行机制。AOT 通常在安装期或系统镜像构建阶段完成，
以降低启动时的即时编译成本；JIT 则在运行期根据方法热度动态生成本地代码，以在不牺牲灵活性的前提下
提升热点路径性能\citep{Cheng2010,ArtJitCompiler2024}。从编译系统角度看，这属于典型的自适应
优化框架：运行时先以较低成本收集热点信息，再将编译资源集中投向最有价值的方法\citep{Arnold2005,
Suganuma2001}。

ART 当前的 JIT 机制并非单级“编了就结束”，而是具有明显的分层特征。方法首先需要达到 warmup 阈值
进入 baseline JIT，在热点继续累积后，再可能触发更高质量的优化重编译；若热点循环在解释执行阶段
已经形成，还可以通过 On-Stack Replacement（OSR）从解释器切换到已编译代码\citep{Fink2003,
Lameed2013}。这种层次化策略的好处在于兼顾了编译时机和机器码质量，但它也引入了新的平台相关问题：
阈值如何设定、代码缓存如何组织、调试信息与编译固定成本如何控制，都会直接影响不同 workload 的最终
收益边界。

移动端 JIT 研究进一步表明，“何时编译、编译到什么程度、是否让不同编译策略协同”本身就是性能优化的
重要部分，而不是编译器后端之外的边角参数\citep{Perez2012HybridJIT,Lee2020,JavART2022}。
因此，在 LoongArch 平台上研究 ART 性能优化时，不能把 JIT 参数调整简单视为工程调参，而应当视为
分层执行机制与具体 workload 生命周期之间的匹配问题。

\subsection{Intrinsic与运行时快速路径}

仅有解释器与编译器主体还不足以支撑成熟运行时。ART 中大量高频 Java 标准库方法依赖 Intrinsic
实现，以便在编译阶段直接映射为更短的机器码序列或调用特定运行时辅助函数；与此同时，对象分配、
字段访问、类型检查、字符串构造、异常抛出和 JNI 调用等路径，也普遍依赖 \texttt{art\_quick\_*}
入口点或相关 trampoline 提供快速服务\citep{AndroidRuntime2024,ArtJitCompiler2024}。

从执行引擎角度看，Intrinsic 与快速路径承担着“缩短热点语义到机器实现距离”的角色。一方面，它们
减少了通用调用路径上的层层间接开销；另一方面，它们也是不同架构平台体现特化能力的重要位置。例如，
字符串相关运行时路径的实现方式会直接影响短字符串密集 workload 的热点分布，而编译器是否能为标准库
方法生成 Intrinsic 代码，则会决定新架构平台在成熟度和性能上的上限\citep{Haubl2008OptimizedStrings}。

需要区分的是，Intrinsic 与普通方法内联并不等价。普通内联仍然以被调方法的中间表示为输入，通过
复制或重写 HIR 后再交给后续优化阶段处理；Intrinsic 则更像一条预先声明的语义捷径：编译器在识别到
特定标准库方法后，可直接为其建立专用的 locations 和 lowering，并发射更短的机器码序列；若当前
架构尚未补齐对应实现，则自动回退到普通 invoke 路径。因此，Intrinsic 的平台适配并不是“多做几条
peephole 优化”，而是在执行引擎中为一组可枚举的库语义建立离散映射表。

从优化对象看，Intrinsic 与循环 SIMD 向量化也分属两个层次。前者主要面向 \texttt{Math}、
\texttt{String}、数组辅助方法以及部分加解密等高频库调用，收益通常体现为缩短调用边界和减少固定
指令序列；后者则主要面向编译器能够识别的显式循环与向量数据流。二者都属于“缩短高层语义到机器实现
距离”的手段，但命中的热点形态并不相同：Intrinsic 更偏向库调用和内建语义路径，循环 SIMD 则更偏向
显式规则数值循环和可识别的数据并行模式。

因此，对新架构平台的 ART 适配而言，补齐 Intrinsic 与运行时快速路径并不是“锦上添花”，而是让
高层 Java 语义真正落到可接受机器级实现上的必要条件。这也是为什么本文后续在移植阶段将其归入执行引擎
能力补齐的组成部分，而在优化阶段又围绕其中若干热点路径继续展开性能研究。

\section{LoongArch架构特性及其对ART的影响}

\subsection{寄存器体系与调用约定}

LoongArch 是面向现代通用处理器设计的 RISC 指令集，采用定长 32 位指令格式，并提供独立的通用寄存器
与浮点寄存器集合\citep{LoongArchISA2023,Hu2023LoongArch}。从 ART 适配视角看，最直接的影响
来自平台调用约定：参数如何在寄存器与栈之间传递，返回值如何编码，多返回类型如何区分，调用者保存与
被调用者保存寄存器如何划分。这些规则会同时作用于解释器入口、JIT 代码生成、JNI 桥接和运行时慢路径
恢复。

与平台无关的中间表示相比，执行引擎真正落地到目标机时，调用约定是最先暴露问题、也是最难绕开的约束。
例如，ShadowFrame 与 native 栈之间如何衔接，\texttt{Thread*}、\texttt{ArtMethod*} 和参数区
在入口时如何布局，浮点返回值是否需要特殊寄存器通道，这些问题都无法通过“平台无关逻辑”自动解决。
因此，在 LoongArch 平台上适配 ART，必须首先建立清晰、稳定且与架构 ABI 一致的寄存器与调用约定
映射关系。

\subsection{控制转移与地址计算特点}

LoongArch 的另一项重要特性，是其控制转移与地址构造方式对代码生成组织有较强影响。由于指令长度固定，
分支位移范围、长短跳转组合、PC 相对寻址以及字面量地址构造都需要在汇编器和代码生成器中显式处理
\citep{LoongArchISA2023}。这对 ART 这样的运行时系统尤其重要，因为解释器派发、慢路径跳转、
链接修补、trampoline 入口和异常回退都大量依赖“先占位、后回填”的控制流组织方式。

在 JIT 后端和运行时入口点中，这类约束会具体表现为三个问题。第一，短分支是否足以覆盖目标块，
决定了慢路径布局方式是否需要调整。第二，字面量、入口点地址和 trampoline 地址的构造方式，
决定了汇编器是否必须提供统一的标签与修补抽象。第三，若平台相关地址计算路径不稳定，则调试信息、
反汇编和崩溃定位都会受到影响。因此，LoongArch 平台的控制转移特点不仅影响“能否发出指令”，
更影响执行引擎整体结构是否易于维护和验证。

\subsection{内存访问、原子操作与同步语义}

ART 运行时在对象分配、垃圾回收、读写屏障、锁实现和线程状态切换中高度依赖内存访问与同步语义。
LoongArch 在普通访存、原子读改写、屏障和特权控制状态等方面给出了明确的体系结构定义
\citep{LoongArchISA2023}。对平台适配而言，这意味着运行时中的对象访问、字段更新、锁字操作以及
部分 GC 协议必须与架构语义保持一致，否则就可能在并发环境下出现极难复现的功能错误。

相比用户态普通程序，ART 对这些语义的敏感度更高。原因在于它既要处理 Java 层对象模型，又要处理
native 层线程和异常状态；一旦同步原语、内存序或异常边界处理不当，问题往往会表现为“偶发崩溃、
偶发错值或偶发性能回退”。因此，LoongArch 的内存与同步语义不是只在锁实现中才重要，而是与执行引擎
正确性和优化安全性同时相关的基础约束。

\section{性能评测与优化问题分析}

\subsection{经典JVM benchmark的代表性}

性能优化研究离不开具有代表性的 benchmark。对于 JVM 运行时研究，DaCapo 与 SPECjvm2008 是最常见、
也最具历史可比性的两类工作负载。前者更接近对象分配、字符串处理和复杂控制流较多的现代 Java 应用，
后者则保留了更偏传统数值与吞吐导向的基准场景\citep{Blackburn2006,SPECjvm2008}。这也是本文后续
选择 DaCapo 的 \texttt{pmd} 与 SPECjvm2008 的 \texttt{mpegaudio} 作为核心观察对象的直接原因。

但基准选择本身也具有方法学边界。后续研究表明，不同 benchmark 套件对优化机会的“可见度”并不一致；
例如 Renaissance 之所以被提出，正是因为传统套件对现代并发与并行工作负载的覆盖不足，难以暴露某些
JIT 优化或运行时机制的真实价值\citep{Prokopec2019Renaissance}。因此，基准结果的解释必须与
workload 特征绑定，而不能把某一子项的收益直接外推为全部应用场景中的普遍结论。

\subsection{Warmup、波动与统计严谨性}

JIT 运行时的性能测量从来不是“跑一次取一个数字”这么简单。由于解释器驻留时间、编译触发时机、
代码缓存状态、GC 行为以及系统噪声的共同影响，Java benchmark 往往具有显著的波动性。统计严谨的
性能评测研究指出，若仅使用 \texttt{best-of-N} 或单次结果，就可能把实验噪声误判为优化收益，甚至
得出方向错误的结论\citep{Georges2007}。

更进一步，warmup 本身并不总是“快速结束并进入稳定峰值”的简单两阶段过程。相关研究表明，不少
JVM 基准即使在严格受控环境下，也可能出现 slowdown、no steady state 等行为；也就是说，丢掉前几轮
数据并不总能得到真实、可比的稳态结果\citep{Barrett2017Warmup}。这对本文尤其重要，因为短生命周期
workload 恰恰更容易让 JIT 阈值、调试信息固定成本和运行时初始化成本放大为主导因素。因此，本文后续
在性能分析中始终强调固定 baseline、重复实验和工作负载边界，而不把单轮结果直接上升为一般性结论。

\subsection{低扰动可观测性需求}

优化研究不仅需要跑分，还需要知道“为什么变快或变慢”。这要求实验系统具备足够的可观测性，包括低扰动
日志、profile、火焰图和必要的运行时统计信息。Android 运行时研究中的 ARTist、Hubble 等工作表明，
若观测机制本身开销过高，测到的现象很可能已经被工具扰动污染；相反，只有把追踪路径压到足够低的成本，
性能调试结果才有解释价值\citep{Luo2022Hubble}。

对 LoongArch 平台 ART 而言，这一点同样成立。由于本文研究对象既包含解释器和 JIT 的时机问题，
又包含运行时字符串热路径等细粒度优化，单看最终 benchmark 分数远不足以解释方案优劣。必须结合
\texttt{run.log}、符号级热点统计、调用图与火焰图等多种证据，才能建立“热点变化 -> 固定成本变化 ->
最终分数变化”的因果链。这也是第 4 章和第 5 章采用统一数据采集目录和固定分析口径的理论基础。

\section{AOSP 15 ART移植的关键技术难点}

\subsection{执行引擎适配范围广且耦合紧密}

从工程边界看，ART 并不是一个单独的解释器程序，而是由解释器、JIT 编译器、JNI 编译器、运行时入口点、
对象模型、调试支撑和测试基础设施共同组成的复杂系统\citep{Wimmer2013,AndroidRuntime2024}。
因此，新架构平台的移植工作天然具有“覆盖范围广、模块之间强耦合”的特征。对 LoongArch 平台而言，
任何一个模块缺失，都可能导致 Java 到 native、解释执行到编译执行、正常路径到异常路径的调用链无法闭合。

这也是为什么既有论文在总结 JVM/ART 移植经验时，往往都采用“由最小闭环逐步扩展”的实现策略，而不是
试图一次性补齐所有高性能路径\citep{Huang2021,Li2019}。本文后续的实现章节同样遵循这一思路：
先让 C++ 解释器成为语义基线，再逐步扩展到 Nterp、JIT、JNI 和快速路径。

\subsection{正确性验证与性能研究必须并行考虑}

对于普通应用程序，编译通过和功能可用往往已经能够说明大部分问题；但对运行时系统而言，编译通过只说明
“接口大致接上了”，并不意味着执行引擎已经具备可验证的正确性，更不意味着其性能结论具备研究价值。
一方面，解释器、JIT 和运行时入口点中的小错误可能只在极端控制流、异常恢复或对象访问边界上暴露；
另一方面，若在存在语义错误的前提下直接进行 benchmark，所有性能结论都会失去解释基础。

因此，新架构 ART 研究必须同时组织正确性验证链路与性能评测链路。前者通过 Host 测试、Target run-test、
libcore 测试等方式证明执行语义收敛；后者则在统一 baseline 和统一工作负载下观察优化收益边界。两者
相辅相成，缺一不可。这也是本文把“系统验证方法与实验平台”单列为一章，而不把所有实验内容混写进移植
实现章节的原因。

\subsection{新版本ART的复杂性与历史可比性问题}

与早期 Dalvik 或早期 ART 相比，AOSP 15 中的执行引擎在解释器实现、JIT 基础设施、调试信息、
运行时组件组织以及测试框架上都更加复杂\citep{AndroidRuntime2024,ArtJitCompiler2024}。
这意味着，旧版本平台移植经验虽然仍有参考价值，但不能机械照搬到新版本源码上。尤其是当源码结构、
运行时策略和性能瓶颈都已发生变化时，研究工作既要保持与既有论文的历史可比性，又必须基于新版本真实
代码路径重新建立问题分析框架。

对本文而言，这种复杂性最终体现为两项要求。第一，移植实现必须以 AOSP 15 当前版本的模块边界和执行
机制为准，而不是以历史版本接口为准。第二，性能优化必须建立在新版本正式 baseline 的热点事实上，
而不是沿用旧版本经验直接选题。只有这样，论文中的“适配”和“优化”两条主线才能同时具有工程真实性与
研究说服力。

\section{本章小结}

本章围绕 ART 的体系结构、解释与编译机制、LoongArch 架构约束、性能评测方法学以及新版本 ART 移植
难点，建立了本文后续研究的技术背景与问题分析框架。归纳而言，LoongArch 平台上的 ART 研究既受到
执行引擎分层结构的约束，也受到目标架构调用约定、控制转移和内存语义的约束；同时，性能结论又必须建立
在统计严谨、低扰动且与 workload 特征一致的实验框架之上。基于这些背景，第 3 章将进一步展开
LoongArch 平台 ART 执行引擎的适配设计与实现。
