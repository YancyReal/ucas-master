\chapter{LoongArch平台ART执行引擎的适配设计与实现}\label{chap:porting}

\section{总体设计目标与实现策略}

相较于普通用户态程序，ART 的平台适配具有更强的系统性。一方面，解释器、JIT 编译器、
JNI 桥接与运行时入口点之间存在紧密耦合关系，任何一个子系统缺失都可能导致 Java 方法
调用链无法闭合；另一方面，ART 在执行过程中高度依赖体系结构相关细节，例如调用约定、
寄存器保存规则、异常现场恢复、分支跳转距离以及线程本地存储访问方式等。因此，本文并未
将 LoongArch 平台适配理解为“把若干汇编文件翻译到新指令集”这一局部任务，而是将其视为
执行引擎整体能力的分阶段构建过程\citep{AndroidRuntime2024,ArtJitCompiler2024,
LoongArchISA2023,LoongArchELFABI2025,LoongArchToolchain2025}。

本文采用由浅入深、由强可移植性路径向高性能路径逐步推进的实现策略。首先，以 C++ 解释器
为起点建立最小可运行闭环。该路径对平台相关组件依赖最少，能够尽早暴露调用约定、栈帧组织、
方法调用和异常传播中的基础问题。其次，在解释器路径可稳定运行的前提下实现 Nterp 汇编解释器，
以逐条补齐 DEX 字节码处理逻辑的方式提高执行效率，并验证 LoongArch 汇编层面的入口、派发与
返回路径。再次，围绕 Optimizing Compiler 补齐 LoongArch64 JIT 后端，使运行时具备从 HIR
到本地机器码的完整转换能力。最后，补齐 JNI 编译器、运行时快速路径和 Intrinsic 支持，使
 LoongArch 平台上的 ART 不仅能够执行 Java 代码，而且能够跨越 Java/native 边界并支撑标准库
热点路径的执行。

这种阶段化实现策略与新架构平台上的 JVM 移植经验是一致的\citep{Huang2021,OpenJDKLoongArch2023}。
其核心思想不是一开始就追求所有执行模式同时可用，而是优先保证最小执行路径闭合，再逐步收敛
高性能路径上的体系结构相关细节。对本文而言，这种策略带来了两个直接收益：其一，问题定位边界
更加清晰，便于将错误限制在单一模块内部；其二，能够尽早建立“可执行、可验证、可回归”的版本，
为后续性能优化提供稳定基线。

\section{汇编器与基础代码生成支持}

\subsection{LoongArch64汇编器实现}

汇编器是新架构适配中最底层、但又最关键的基础设施。无论是 Nterp 汇编解释器、JIT 代码生成、
JNI 宏汇编器，还是运行时入口点桩代码，最终都需要依赖统一的机器指令发射接口。因此，本文首先
完成 LoongArch64 汇编器、寄存器描述与反汇编支持的建设，使其能够支撑上层组件持续演进。

LoongArch64 采用定长 32 位指令格式，寄存器体系和立即数字段划分与 ARM64、x86\_64 均有明显差异
\citep{LoongArchISA2023,LoongArchELFABI2025}。
为了在 ART 中形成统一抽象，本文首先补齐了 LoongArch64 的通用寄存器、浮点寄存器和寄存器打印
接口，使寄存器分配器、调用约定访问器和调试工具能够共享同一套机器描述。随后，围绕汇编器发射
路径逐步实现了整数运算、访存、跳转、屏障、原子操作、浮点运算、移位、时间读取和常用伪指令等
指令族。其中，立即数加载、大偏移访存和 PC 相对寻址被优先实现，因为它们同时影响解释器派发、
JIT 字面量加载和运行时桩代码生成。

与只面向手写汇编文件的工具链不同，ART 中的汇编器还需要服务于编译器自动生成代码。因此，
本文不仅实现了基础指令编码，还提供了适用于编译器场景的组合式接口，例如任意 32/64 位常量加载、
大偏移地址加法、基于标签的分支占位与回填等。这些接口将 LoongArch64 指令编码细节封装在
\texttt{AssemblerLOONGARCH64} 内部，使上层代码生成器可以更关注语义而不是位域拼装。

在汇编器基本可用之后，本文进一步补齐了 LoongArch64 反汇编器与配套测试。前者用于在崩溃定位、
代码生成验证和 CFG 导出中提供可读汇编；后者则通过逐条比对输出结果验证编码正确性，从而保证
后续 Nterp、JIT 和运行时入口点能够建立在可靠的底层发射能力之上。

\subsection{分支、字面量与链接修补机制}

在执行引擎适配过程中，分支与地址修补机制的重要性并不低于普通算术指令。原因在于，ART 的
解释器和编译器大量使用标签跳转、异常慢路径跳转、延迟绑定入口点以及字面量池访问，这些都要求
汇编器在代码发射时允许“先占位、后回填”的增量式组织方式。

LoongArch64 的控制转移指令在跳转距离上具有明确限制，短分支、长跳转和 PC 相对地址构造的编码
方式彼此不同\citep{LoongArchISA2023,LoongArchToolchain2025}。为此，本文在汇编器与代码生成器中实现了标签绑定、字面量
记录和链接修补三类机制。其核心思想是：在第一次发射分支时仅记录语义信息和占位编码；待目标标签
或字面量地址确定后，再在代码整理阶段统一计算真实位移并选择合适的实现形式。这一机制同时服务于
如下三个场景。

第一，服务于 Nterp 派发表与辅助函数跳转。解释器中的公共 slow path 和若干 helper 例程需要依赖
跨块跳转，如果缺乏统一的标签与分支回填机制，将很难在指令尺寸受限的前提下保持代码可维护性。

第二，服务于 JIT 生成代码中的 SlowPath 和异常路径。诸如空指针检查、边界检查、类初始化检查等
路径通常采用“主路径短跳转 + 慢路径延后布局”的组织方式。本文实现的链接修补能力使得
\texttt{CodeGeneratorLOONGARCH64} 可以按语义组织控制流，而将最终分支编码选择延迟到
\texttt{FinalizeCode} 阶段。

第三，服务于运行时入口点和 trampoline 代码。方法调用、接口冲突表、解析入口点和 deopt 入口往往
带有平台特定的长跳转需求。通过统一的标签、字面量和链接修补抽象，本文避免了在不同模块中重复实现
同类地址计算逻辑，提高了执行引擎各子系统之间的一致性。

\section{解释器适配}

\subsection{C++解释器适配}

C++ 解释器是本文建立 LoongArch 平台最小可运行路径的起点。与 Nterp 和 JIT 相比，C++ 解释器
对平台特定汇编能力的依赖最少，其核心执行循环主要由平台无关的 \texttt{switch-case} 逻辑构成，
因此能够尽早暴露方法调用、参数传递、ShadowFrame 布局以及异常传播中的底层问题。

尽管解释器主体使用 C++ 编写，但从 \texttt{ArtMethod::Invoke()} 进入解释执行仍涉及多项平台相关
内容。首先，需要保证 LoongArch64 调用约定与 ART 的栈帧组织方式一致，使解释器入口可以正确接收
\texttt{Thread*}、\texttt{ArtMethod*}、参数区和返回值位置等关键信息。其次，需要补齐与解释器
配套的汇编辅助路径，例如静态/直接调用桩代码、方法返回寄存器恢复和部分运行时入口。再次，需要
确保 DEX PC、调用现场和异常状态能够在 LoongArch64 上被正确保存与恢复，否则即使简单方法能够
执行，异常处理和调试也无法稳定工作。

在实现顺序上，本文首先保证最简单的 Java 程序能够走通“方法调用、解释执行、结果返回”的闭环，
随后逐步补齐对象分配、字段访问、类型检查和异常处理相关路径。这样做的目的不是长期停留在低性能
解释执行，而是借助最可移植的执行模式为后续 Nterp 和 JIT 提供语义基线。换言之，C++ 解释器在
本文中既是功能起点，也是后续高性能路径的重要参照系。

\subsection{Nterp框架建立}

在 C++ 解释器具备基本可运行能力后，本文继续实现 LoongArch64 下的 Nterp 汇编解释器。Nterp
是 ART 中面向热点解释执行的高性能路径，其基本思想是通过预先组织好的派发表，将 opcode 直接映射到
对应的汇编处理逻辑，从而减少传统 \texttt{switch-case} 解释器中的分支分派开销。由于该路径高度依赖
平台汇编实现，因此也是新架构适配中最能体现体系结构细节的部分。

LoongArch64 Nterp 的建立首先需要定义专用 ABI 约束，包括线程寄存器、解释器帧指针、当前 DEX PC、
当前指令缓存和派发表基址等关键寄存器的固定用法。与普通函数调用不同，Nterp 的目标是在解释器主循环
内部尽量避免多余保存/恢复开销，因此寄存器分工必须在入口处一次性建立，并在后续 opcode 处理逻辑中
保持稳定。为此，本文首先实现 Nterp 框架骨架，包括入口函数、栈帧建立、参数装载、类初始化路径以及
与 C++ 解释器之间的共享帧布局。这种围绕固定寄存器协议和直接分派组织解释器热路径的做法，与高效解释器
研究中对分派结构和寄存器约束的结论是一致的\citep{Ertl2003,LoongArchELFABI2025}。

在入口建立之后，本文进一步补齐 Nterp 的派发表组织方式。具体而言，LoongArch64 平台通过“固定宽度
处理块 + 派发表基址”构造字节码到汇编代码的快速映射关系，使常见指令能够以较低分派成本进入对应处理
例程。对于超出单个处理块容量或需要复杂逻辑的 opcode，则通过 helper 和 slow path 方式进行扩展。
这种组织方式一方面保证了解释器主路径紧凑，另一方面也使复杂语义可以在不破坏整体结构的前提下逐步补齐。

\subsection{Nterp指令补齐与问题修复}

Nterp 的难点并不在于建立入口骨架，而在于将 200 余条 DEX 字节码处理逻辑逐步补齐，并在这一过程中
保证与 C++ 解释器完全一致的语义。本文在实现时采用“按指令类别逐步闭环”的方式推进：首先补齐常量、
返回、移动和整数算术等基础指令，使最小测试用例可以运行；随后继续实现浮点运算、静态字段访问、实例
字段访问、数组访存、对象创建、类型检查和类初始化相关指令；最后补齐各类 invoke 及其 range 变体，
并引入 nterp-to-nterp 快路径和必要的 slow path。

这一迭代过程中，本文集中解决了若干具有代表性的 LoongArch64 平台问题。其一，Nterp 与调用桩代码
之间的栈帧衔接问题。如果解释器栈与 native 栈对齐规则不一致，方法调用返回时就可能发生栈破坏，
进而导致 invoke 返回路径崩溃。本文通过统一对齐策略和保存/恢复顺序，修复了 invoke stub 返回后的
栈破坏问题。其二，浮点 NaN 分类问题。LoongArch64 在浮点比较和分类上的表现与既有平台存在差异，
若直接沿用其他架构的判断路径，可能导致部分浮点测试出现不一致。本文针对该问题补充了平台化修正。
其三，JIT 与 Nterp 混合执行时的对象状态问题。随着 JIT 路径逐步可用，Nterp 不再是孤立执行路径，
其与编译代码之间的对象与寄存器状态切换必须保持一致。本文进一步修复了 JIT invoke 场景下的对象损坏
问题，确保解释执行与编译执行能够在同一运行时环境中稳定共存。

通过上述逐类补齐与问题修复，LoongArch64 Nterp 最终从“仅能运行极简样例”的骨架逐步演化为能够覆盖
完整 DEX 字节码集的汇编解释器。这一结果意味着 LoongArch 平台不仅具备了平台无关的解释执行能力，
也具备了与主流架构接近的高性能解释执行通路。

\section{JIT编译器后端适配}

\subsection{LoongArch64代码生成器实现}

如果说解释器适配解决的是“代码尚未编译时如何执行”的问题，那么 JIT 后端适配解决的就是“热点代码
如何生成本地机器码”的问题。ART 的 Optimizing Compiler 以 HIR 为核心中间表示，前端负责构建
\texttt{HGraph} 并执行平台无关优化，平台后端则负责寄存器分配、调用约定映射和机器指令发射。
因此，LoongArch64 JIT 后端的核心任务，是将 ART 已有的优化编译基础设施同 LoongArch64 的寄存器、
指令和调用约定约束连接起来\citep{Aho2006,Wimmer2013,LoongArchELFABI2025,
LoongArchToolchain2025}。

本文围绕这一目标，首先补齐了 LoongArch64 的代码生成器类、寄存器阻塞规则、调用约定访问器和栈帧
管理逻辑。具体而言，\texttt{CodeGeneratorLOONGARCH64} 负责统一组织代码发射，
\texttt{InstructionCodeGeneratorLOONGARCH64} 负责按 HIR 指令生成平台相关机器代码，
调用约定访问器则负责在 Java 调用、运行时调用和返回值传递之间建立一致的寄存器/栈映射关系。
在这一基础上，本文进一步实现了并行移动、参数装载、返回寄存器恢复、帧建立与销毁、显式/隐式空指针
检查以及若干辅助工具接口，使 LoongArch64 后端可以接入 ART 的寄存器分配与代码布局流程。

\subsection{核心HIR节点支持}

JIT 后端适配的主体工作，是将平台无关 HIR 节点映射为可执行的 LoongArch64 指令序列。本文按照从
基础到复杂的顺序逐步补齐相关节点支持。首先，完成整型/长整型/浮点常量、算术运算、移位、比较、
分支、返回和参数节点等基础 HIR，使简单方法具备基本编译执行能力。随后，继续补齐数组长度、
边界检查、字段读写、类加载、异常值装载和方法句柄等更接近真实应用的节点，以支持常见对象与数组操作。
最后，围绕静态调用、直接调用、虚调用、接口调用、未解析调用以及自定义/多态调用等复杂语义补齐
invoke 族节点，使 LoongArch64 后端能够覆盖 ART 中最关键的控制流与调用场景。

在这一过程中，本文特别重视“主路径代码生成”和“慢路径处理”之间的协同组织。对于空指针、越界、
类初始化和异常抛出等语义，JIT 后端并不直接在主路径中展开复杂处理，而是采用主路径短检查加慢路径
回退的方式降低正常执行开销。为实现这一点，本文在 LoongArch64 后端中配套实现了必要的 linker patch、
literal 访问、运行时调用封装和 SlowPath 对象布局机制，使热点路径与异常路径能够在统一框架下共存。

\subsection{OSR与运行时协同支持}

JIT 后端闭环不仅要求“能生成代码”，还要求所生成代码能够与运行时共同维护热度、去优化和异常恢复等
动态行为。为此，本文进一步补齐了 On-Stack Replacement（OSR）、去优化、方法进入/退出钩子以及
内联缓存更新相关支持\citep{Fink2003,Lameed2013}。其中，OSR 允许热点循环在执行过程中从解释器
切换到已编译代码，是 ART 分层执行机制中的关键环节；去优化则保证在假设失效或运行时状态变化时，
已编译代码可以安全回退至解释执行。

在实现层面，本文为 LoongArch64 平台接入了必要的运行时入口和协同逻辑，包括热度递增、内联缓存更新、
去优化入口、字符串构建辅助、快速方法调用和崩溃定位所需的反汇编能力等。这样一来，LoongArch64 JIT
后端不再只是“静态代码发射器”，而是能够真正参与 ART 的动态执行生态，在解释执行、编译执行与运行时
状态切换之间保持一致行为。

\section{JNI编译器与运行时快速路径实现}

\subsection{JNI调用桥接实现}

在 ART 中，Java 代码与 native 代码之间的调用并不直接遵循普通 C 函数调用语义，而是需要经过
专门的 JNI 桥接层完成线程状态切换、参数整理、栈帧组织与异常传播。对于新架构平台而言，若这一桥接
层没有完整实现，即使解释器与 JIT 都已可用，Java 方法也仍无法稳定调用 native 代码。因此，JNI 编译器
和 JNI 宏汇编器是执行引擎闭环中不可缺少的一环。

本文首先实现了 LoongArch64 JNI 调用约定描述，使 Java 参数、返回值、\texttt{JNIEnv*} 与隐藏参数
可以按平台规则在寄存器和栈之间分配。随后，以此为基础补齐了 JNI macro assembler 骨架，并逐步实现
保存现场、构建 JNI 帧、线程状态从 Runnable 切换到 Native 以及从 Native 返回 Runnable 的核心路径。
在宏汇编器稳定后，本文进一步接入普通 native 方法的 JNI 编译支持，并补齐栈展开相关的 \texttt{WriteCIE()}
需求，使 LoongArch64 平台能够在异常与调试场景下正确恢复调用链\citep{LoongArchELFABI2025}。

需要指出的是，JNI 路径的适配原则并非一开始就追求最激进的 native 快路径。本文在适配阶段优先保证
普通 JNI 调用与基本 native 方法执行闭环，对更激进的 direct \texttt{@CriticalNative} 路径则采取保守
门控策略。这样做的原因在于，该路径对调用约定、异常处理和返回值封装一致性要求更高；在平台化适配
尚处于收敛阶段时，优先保证功能正确性比贸然放开高风险快路径更为稳妥。

\subsection{运行时入口点与快速路径补齐}

JIT 和解释器都高度依赖运行时入口点提供服务。对象分配、锁操作、字段访问、类解析、类型检查、
异常抛出和方法调用等高频操作，最终都需要通过 \texttt{art\_quick\_*} 入口点或相关 trampoline 落到
运行时实现上。因此，运行时快速路径的补齐并非附属工作，而是 LoongArch64 执行引擎真正可用的必要条件。

围绕这一目标，本文逐步实现和接入了对象分配入口点、\texttt{CheckInstanceOf} 相关入口、快速字段访问
入口、快速方法调用桩、类解析入口、字符串相关入口、deopt 入口以及数学辅助入口等关键组件。与此同时，
本文还补齐了与方法分派和异常恢复相关的支撑模块，例如 \texttt{ImtConflictTable} trampoline、
\texttt{SignalHandler} 和部分故障恢复路径，使编译代码和解释器在面对接口冲突、信号异常和运行时错误时
均能够回到受控处理流程。

从工程角度看，这部分工作的价值在于将“看似零散的架构相关函数”组织成统一的运行时支撑层。只有当
解释器、JIT 和 JNI 都能够共享这一套运行时入口点时，LoongArch 平台上的 ART 才具备与主流架构相同的
执行语义基础。

\section{Intrinsic与执行引擎能力扩展}

在完成解释器、JIT 和 JNI 主体闭环后，LoongArch 平台仍面临一个容易被忽视但实际非常关键的问题：
即 Java 标准库中的热点方法在新平台上是否具备足够的执行支持。ART 中大量热点方法依赖 Intrinsic
实现，以便在编译阶段直接映射为更短、更高效的机器代码。若新架构缺失这些支持，虽然程序仍可通过常规
调用路径运行，但在性能和兼容性上都会与成熟平台存在明显差距。

为此，本文进一步实现并扩展了 LoongArch64 相关 Intrinsic 与运行时能力，包括 Memory peek/poke、
整型/浮点最值计算、\texttt{Float} 与 \texttt{Double} 相关 Intrinsic，以及 \texttt{Math.fma}
等典型数值计算路径。与此同时，本文还根据 AOSP 15 的演化情况同步扩展了运行时支撑，修复和补齐了
若干原本仅在其他架构上完整可用的热点路径。

需要强调的是，本文在适配阶段对 Intrinsic 的处理原则是“优先补齐能力，再讨论进一步优化”。也就是说，
这一阶段的目标首先是确保标准库热点路径在 LoongArch 平台上具备正确、稳定的机器级实现，而不是立即将
所有平台特化收益都压榨到极致。这一保守而渐进的策略，为后续在稳定基线之上继续开展执行引擎优化提供了
更可靠的起点。

\section{阶段性结果分析}

从阶段结果来看，本文的适配工作呈现出明显的逐层闭环特征。最初，LoongArch 平台仅具备零散的机器描述和
部分汇编能力，尚不足以支撑 Java 方法执行。随着汇编器、C++ 解释器和基础运行时路径补齐，系统首先具备了
最小可运行能力。随后，Nterp 从骨架逐步扩展到完整字节码覆盖，解释执行路径从“能跑通样例”提升到
“可承担主力解释执行”。在此基础上，JIT 后端、JNI 编译器和运行时快速路径相继闭合，使编译执行与
Java/native 调用链也具备了可用性。最终，通过 Intrinsic 和若干保守门控策略的补充，LoongArch 平台上的
ART 执行引擎形成了可用于系统验证与后续性能研究的稳定基线。

从研究对象边界看，本文将从初始移植起点到稳定基线版本这一提交区间视为“执行引擎适配阶段”的主体成果。
这一阶段工作的核心价值，不在于单个局部优化点，而在于系统性补齐了 LoongArch 平台上 ART 执行引擎所需的
关键构件，使后续的正确性验证和性能优化建立在完整、可复现的工程基础之上。

\section{本章小结}

本章围绕 LoongArch 平台 ART 执行引擎适配，系统介绍了汇编器与基础代码生成能力、C++ 解释器与
Nterp 汇编解释器、JIT 编译器后端、JNI 调用桥接、运行时快速路径以及 Intrinsic 扩展等方面的实现工作。
这些工作共同构成了新架构平台上 ART 的完整执行基础。其结果表明，只有将解释执行、编译执行、Java/native
桥接与运行时支撑作为一个整体来适配，LoongArch 平台上的 ART 才能从“局部可运行”演进为“系统可验证、
可优化”的完整运行时。下一章将在此基础上进一步给出正确性验证方法、实验平台组织方式以及正式 baseline
的建立过程。
